# Niko Drone Simulator - AI 開發核心導航文件

## 1. 系統本質與定位
*   **軟體本質**：本專案為專業無人機「模擬器」，所有通訊與邏輯設計均服務於「虛擬操控與考照訓練」。
*   **核心目標**：在 Android 行動裝置上提供高精度、流暢且符合法規標準的虛擬飛行體驗。

## 2. 硬體架構特性 (關鍵)
*   **內置非 HID 搖桿支援**：專案重點在於支援具備內置搖桿的專業 Android 遙控器（如 RadioMaster, MK15, AX12 等）。
*   **RX 偵測定義**：
    *   **非**真實無人機通訊。
    *   **是指**從底層串列總線 (Internal Serial Bus) 抓取原始封包 (Raw Packets) 的程序。
    *   目的在於完成「底層封包通道」到「模擬器虛擬軸 (Joystick Axis)」的**映射綁定**。
*   **通訊協議**：主要透過 Serial (UART) 讀取數據，常見協議包含 UMBUS, SBUS, IBUS 等。

## 3. 開發禁令與原則
*   **禁止錯誤類比**：切勿將模擬器內的連線邏輯與真實無人機的 RF 失控保護 (Failsafe) 混淆。
*   **映射優先**：所有通訊優化必須以提升「映射精確度」與「硬體相容性」為最高準則。
*   **低延遲要求**：內置搖桿的封包解析必須維持在最高優先級，確保操作手感無延遲。
*   **不要亂加版號**：所有的版號以AppConfig.kt裏的版號為基礎。

## 4. 模組職責劃分
*   `UsbSerialManager`: 負責底層物理連線與原始數據流抓取。
*   `InputProcessor`: 負責將解析後的數值轉換為飛行指令。
*   `DroneViewModel`: 負責業務邏輯統籌（如引導教學、狀態監控）。
*   `FieldRenderer`: 負責 3D 地景與地面標線渲染。
*   `DroneHUD`: 負責 2D 飛行數據與 UI 提示。

# AI 程式編寫規範手冊

這是一份旨在減少大型語言模型（LLM）常見程式碼錯誤的行為準則。請根據具體專案需求進行調整。

**權衡取捨：** 這些準則偏向「謹慎」而非「速度」。若處理瑣碎任務，請自行斟酌。

## 1. 動手開發前先思考

**不預設、不隱瞞疑慮、明確點出權衡點。**

在實作之前：
- **明確說明假設**：如果不確定，請發問。
- **呈現多種方案**：若存在多種詮釋方式，請條列說明，不要擅自決定。
- **推動簡化**：若有更簡單的作法，請提出。必要時請推動更優方案。
- **及時止步**：若有不明確之處，請停下來並說明困惑點，然後發問。

## 2. 簡潔至上

**用最精簡的程式碼解決問題。不加入任何投機性的開發。**

- **拒絕過度開發**：不開發需求以外的功能。
- **避免過早抽象化**：不為僅使用一次的程式碼進行抽象化（Abstractions）。
- **不預留靈活性**：不加入未經要求的「靈活性」或「可配置性」。
- **不做多餘處理**：不為不可能發生的場景編寫錯誤處理（Error handling）。
- **精簡邏輯**：如果你寫了 200 行，但其實 50 行就能搞定，請重寫。

> **自我檢視：** 「資深工程師會覺得這太過複雜嗎？」如果是，請簡化。

## 3. 精準修改

**只動必要的程式碼。只清理自己造成的改動。**

編輯既有程式碼時：
- **不隨意優化**：不要「順便優化」鄰近的程式碼、註解或格式。
- **不重構穩定代碼**：不要重構沒壞掉的東西。
- **遵循既有風格**：遵循既有的風格，即使你有不同的偏好。
- **告知廢棄代碼**：若發現與此次修改無關的廢棄程式碼（Dead code），請口頭告知，不要直接刪除。

當你的修改導致產生冗餘時：
- **清理關聯變動**：刪除因「你的修改」而不再使用的引用（Imports）、變數或函式。
- **保留既有冗餘**：除非被要求，否則不要刪除原本就存在的廢棄程式碼。

**測試基準：** 每一行變更都必須能直接溯源至使用者的需求。

## 4. 目標導向執行

**定義成功準則。不斷循環直到驗證通過。**

## 5. Niko Project Specifics (Niko 專案專屬執行細則)

當在 **Niko Drone Simulator** 專案環境中工作時，必須額外遵守以下「外科手術式」執行規範：

### 5.1 任務隔離與禁止自作主張
- **精確執行**：僅修改與當下任務直接相關的邏輯。嚴禁為了優化而改動相鄰程式碼、註解或排版。
- **僅限報告 (Report Only)**：若執行中發現其餘 Bug、效能問題或代碼壞味道（Code Smell），**嚴禁直接修改**。請在回覆末尾建立 `【潛在問題報告】` 區塊進行條列說明。
- **預設參數鎖定**：
    - **[REFERENCE_ONLY]** 系統預設 `ExpoValue` 為 `0.4`。
    - 除非明確要求，否則直接以常數實作，禁止設計複雜的配置管理或微調介面。

### 5.2 硬體通訊與效能紅線
- **通訊協議假設**：在處理 UMBUS 或序列埠掃描前，必須先陳述假設。涉及 `/dev/tty` 節點時，若發現權限風險（如 Android 9+ SELinux 限制），必須先報告並提出替代方案。
- **效能優先**：所有 Input 處理與 RX 偵測邏輯必須位於非同步線程。若修改可能導致渲染掉幀（Frame Drop），必須立即停止並回報。

### 5.3 驗證標準
- 修改後必須確認：
    1. 該改動是否能追溯至原始請求？
    2. 是否有任何非必要的自動重構或格式調整？（若有，請復原，驗證回復內容後需報告）。
    3. 觀察到的其餘問題是否已正確移至報告區而非直接修正？

### 5.4 註解、備份與跳過規範 (後悔藥機制)
- **忽略標籤**：凡是包裹在 `` 標籤內，或行首標記為 `[REFERENCE_ONLY]`、`[ARCHIVED]`、`[DEPRECATED]` 的內容，Agent 在分析專案邏輯或產出代碼時必須**予以忽略**。
- **用途定義**：上述內容僅視為開發者的私人註解、歷史備份或未來可能的備選方案，不代表目前的系統現況或指令。
- **禁止調用**：除非開發者明確要求「恢復 [REFERENCE_ONLY] 中的邏輯」，否則不得將其作為代碼實作的依據。

### 5.5 UI Engineering & Layout Integrity (UI 佈局穩定性規範)
為了防止在現代佈局引擎下發生測量死鎖或內容塌陷，所有 UI 開發必須遵守以下「防呆」標準：

1. **繪製與排版隔離 (Drawing vs Layout Separation)**：
    - **規則**：所有裝飾性、提示性功能（如：捲動箭頭、邊框發光、背景漸層）**嚴禁**使用 `Box`、`Column`、`Text` 等實體元件實現。
    - **標準做法**：必須使用 **`Modifier.drawWithContent { ... }`** 直接在 Canvas 層繪製。
    - **目的**：確保提示效果不參與 Compose 的 Layout 測量階段，100% 捍衛佈局穩定。

2. **徹底杜絕「嵌套權重死結」 (Nested Weight Audit)**：
    - **規則**：當外部容器具備無限延伸屬性（如 `verticalScroll`）時，內層子元件**絕對禁止**使用 `Modifier.weight()`。
    - **標準做法**：使用 **`fillMaxWidth()`** (寬度固定) 與 **`wrapContentHeight()`**。若需比例對齊，改用 **`fillMaxWidth(0.5f)`** 等顯式比例，或 **`height(IntrinsicSize.Min)`**。
    - **目的**：防止測量引擎因高度無限而導致權重計算歸零（畫面消失）。

3. **全自動溢出感應範式 (Automatic Overflow Detection)**：
    - **規則**：禁止手動硬編碼「是否顯示捲動提示」。
    - **標準做法**：統一監測 **`scrollState.maxValue > 0`** 作為判斷內容是否溢出的唯一金標準。

4. **交互層級的「零攔截」保證 (Non-blocking Interaction)**：
    - **規則**：裝飾提示層絕不能影響底層按鈕、滑桿的核心操作。
    - **標準做法**：裝飾層必須透過 **`DrawScope`** 實現，或顯式標記 **`pointerInput(Unit) {}`** (不攔截點擊)。

**成效指標：**
若符合以下情況，代表這些準則運作良好：
* 差異比對（Diffs）中不必要的變動變少。
* 因過度複雜而重寫的情況減少。
* 在實作前而非出錯後提出澄清問題。
